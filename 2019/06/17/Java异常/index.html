Java—-异常

异常的处理机制

使用try…catch捕获

如果try里的代码块出现异常,系统会自动生成一个异常对象,该异常将会throw给运行时对象。

当java运行时环境接受到异常时，会寻找处理该异常的catch块，如果找到合适的catch。如

果没有找到合适的catch块，运行就会终止，java程序就会退出。

异常的继承体系

首先异常子类都有共同的父类：Exception，所有的异常都要直接或者间接继承此类。异常分

为两种类型，一个是编译时（Checked）异常，一个是运行时（RuntimeException）异常。

Java接受到异常后会检测是否为catch块中的异常的子类或实例，如果是就会调用catch块来

执行，如果不是就会将其与下一个catch块中的异常进行比较，这就看出

**当try后有多个catch块出现时，就需要将子类的catch块放在前面，否则就会被父类的异常

直接屏蔽。**

try和catch后面的{}不能省略，而且try中的变量属于局部变量，不能在外面直接使用。

使用finally回收资源

Java垃圾回收机制不会回收任何的物理资源，只会回收堆内存中的对象占用的内存。因此使

用finally块可以解决。不管try中的代码是否出现异常，也不管catch中的代码是否被执行，

甚至在try或catch中出现return语句，finally也将被执行。其中try时必须的，而catch和f

inally必须存在在try后面。但catch和finally中必须要出现一个。finally必须出现在最后

。但是当在finally出现了System.exit();finally将失去执行的机会。

Checked和Runtime异常体系

所有的Runtime子类被称为Runtime异常。Check则是java认为的能被修复的异常，所以必须显

式的处理它，否则就会编译时异常。Runtime类则无需显式处理，

捕获同样可以通过try catch。

throws声明抛出异常

使用throws声明抛出思路：

当前方法不知道如何处理这种异常，该异常应该由上一级调用者处理，如果main方法也不知

道如何处理，可以使用throws抛出，将该异常交给jvm处理，jvm将会终止程序的运行。

使用throws后将不再需要try catch来捕获了，当某段代码调用了一个throws的方法，该方法

就希望它的调用者来处理该异常。也就是调用该方法时，要么放在try语句中显式捕获，要么

放在另一个带throws抛出的方法中。

使用throw抛出异常

由于异常的主观性较强，所以java可以通过自定义异常类来抛出异常，而这就需要throw。使

用throw抛出的是一个异常的实例，而不是一个异常类，且一次只能抛出一个异常实例。通常

在用try catch中，使用if等选择分支判断，然后使用throw来抛出一个异常类的实例。当满

足类if的条件并抛出了一个异常实例时，java就会终止当前的执行流，并找到对应的catch来

处理，自定义异常类必须继承Exception或它的子类，定义一个自定义的异常类通常需要两个

构造器

当一个异常出现后，无法单靠个方法处理该异常时，就需要几种方法协作处理，这样可以把

throw写在catch块中，这样系统就会再次跳到另一个异常中处理。

try catch throw throws流程

首先写一个自定义的异常类，写一个能够throw的方法，在判断条件的不成立的后面用throw，

throw的用法是thrownew 异常类类名，在方法名后面写一个throws用来抛出这个异常给调用

者。此时一旦调用这个方法就必须要将其放在try 块内部，而后面必须有一个用来接收该自

定义异常类的catch块，（catch块写printStackTrcace方法，该方法的作用是打印出自定义

异常中想要打印的信息。打印的信息是由throw出的自定义异常类的对象的参数传入的。调用

者是该try中抛出的自定义异常类的对象。该方法继承自父类Exception。）否则就会将该异

常抛给main方法。虚拟机将会停止。
